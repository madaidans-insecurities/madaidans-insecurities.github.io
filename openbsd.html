OpenBSD

OpenBSD is lacking in a lot of ways. Many of its exploit mitigations are half-baked/useless such as its W^X which can be bypassed by transitioning a writable memory mapping to executable or TRAPSLED which doesn't actually make exploitation harder in the era of ASLR.

OpenBSD is also missing many important security features such as Control-Flow Integrity, SafeStack or ShadowCallStack (RETGUARD is not a replacement for CFI or SafeStack/ShadowCallStack since it only covers backward edges and is bypassed with a single information leak), verified boot, Trusted Path Execution and a lot more. OpenBSD hasn't innovated much and many of its (real) mitigations were developed long before OpenBSD implemented them such as W^X and ASLR which were developed by PaX (PAX_MPROTECT and PAX_ASLR).

Although, some of OpenBSD's mitigations are pretty solid like their malloc implementation.

OpenBSD has a non-existent application security model. It does not have any MAC implementation to fully lock down userspace and lacks any GUI isolation. It uses Xenocara, a fork of X11 and does not support Wayland.

OpenBSD has received criticism from many respected security experts. One example is Brad Spengler, the developer of grsecurity.

This website goes in-depth into the issues with most of OpenBSD's mitigations: https://isopenbsdsecu.re/

A more promising alternative to OpenBSD is HardenedBSD. There are advantages/disadvantages of HardenedBSD over OpenBSD. HardenedBSD has many mitigations OpenBSD does not such as CFI, SafeStack, SEGVGUARD, a proper W^X implementation and more but it lacks things OpenBSD has such as LibreSSL, a hardened malloc etc.
